# AR_002.5_D — Schema: match_analytics_cache para Relatórios V1.1

## Descrição
LACUNA DETECTADA: O PRD Seção 4.2 lista 'Relatórios Avançados: analytics de match events' como entregável V1.1. O schema atual possui training_analytics_cache (para treinos, com 17 métricas em colunas dedicadas), MAS não possui equivalente para partidas. Calcular em tempo real: gols/atleta, arremessos/atleta, eficiência por jogo, defesas/goleira, turnovers/atleta requer full scan de match_events a cada requisição — inaceitável para o SLA de < 2s (PRD Seção 5, Fase 2 métricas).

AÇÃO: Criar migration Alembic para a tabela match_analytics_cache. Usar Python (.py).

ESTRUTURA DA TABELA:
```sql
CREATE TABLE public.match_analytics_cache (
    id uuid DEFAULT gen_random_uuid() NOT NULL,
    match_id uuid NOT NULL REFERENCES matches(id) ON DELETE CASCADE,
    team_id uuid NOT NULL REFERENCES teams(id) ON DELETE RESTRICT,
    athlete_id uuid REFERENCES athletes(id) ON DELETE SET NULL,  -- NULL = stats do time
    cache_type character varying(32) NOT NULL,  -- 'team' | 'athlete' | 'goalkeeper'
    -- Campos de arremesso e gol
    total_shots integer DEFAULT 0 NOT NULL,
    total_goals integer DEFAULT 0 NOT NULL,
    goals_7m integer DEFAULT 0 NOT NULL,
    shot_conversion_pct numeric(5,2),  -- total_goals / total_shots * 100
    -- Campos de goleira
    total_saves integer DEFAULT 0 NOT NULL,
    goals_conceded integer DEFAULT 0 NOT NULL,
    goalkeeper_efficiency_pct numeric(5,2),  -- saves / (saves + goals_conceded) * 100
    -- Disciplinar
    yellow_cards integer DEFAULT 0 NOT NULL,
    red_cards integer DEFAULT 0 NOT NULL,
    exclusions_2min integer DEFAULT 0 NOT NULL,
    -- Outros
    total_turnovers integer DEFAULT 0 NOT NULL,
    total_assists integer DEFAULT 0 NOT NULL,
    -- Cache metadata
    computed_at timestamptz DEFAULT now() NOT NULL,
    is_final boolean DEFAULT false NOT NULL,  -- true quando match.status='finished'
    CONSTRAINT ck_match_analytics_cache_type CHECK (cache_type IN ('team', 'athlete', 'goalkeeper'))
);
CREATE UNIQUE INDEX ux_match_analytics_cache_match_team_athlete ON match_analytics_cache(match_id, team_id, COALESCE(athlete_id, '00000000-0000-0000-0000-000000000000'::uuid), cache_type);
CREATE INDEX ix_match_analytics_cache_match_id ON match_analytics_cache(match_id);
```

REGRA DE ATUALIZAÇÃO: Celery task após cada INSERT em match_events que recalcula e upserts o cache. Quando match.status='finished', is_final=TRUE e o cache não é mais atualizado.

## Análise de Impacto
- **Escopo (atômico):** criação de uma migration Alembic para materializar a tabela `match_analytics_cache`, seus constraints e índices, sem alterar regras de negócio fora do cache.
- **Numeração da migration:** como a cadeia atual já está em `0058`, a nova migration será `0059` com `down_revision='0058'`.
- **Modelo de dados:** tabela de cache por `match_id + team_id + athlete_id + cache_type`, com suporte a agregações de time e atleta (athlete_id nullable).
- **Integridade:** FK para `matches` (ON DELETE CASCADE), `teams` (ON DELETE RESTRICT) e `athletes` (ON DELETE SET NULL), além de `CHECK` para `cache_type`.
- **Índices:** 1 índice simples por `match_id` e 1 índice único funcional usando `COALESCE(athlete_id, uuid_nil)` para impedir duplicidade entre linhas de time/atleta por tipo.
- **Riscos principais:** colisão de nome/revision no histórico Alembic e possíveis incompatibilidades de sintaxe SQL de índice funcional; mitigado por revisão da cadeia atual e criação explícita via SQL.
- **Rollback:** `alembic downgrade -1` (remove tabela e índices associados).
