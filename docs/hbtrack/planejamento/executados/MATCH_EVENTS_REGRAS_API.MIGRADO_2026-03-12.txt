## MATCH EVENTS REGRAS API

1. **O cliente não é dono do `matchEventId` canônico.**
   Na superfície sync, o cliente envia `requestId` e, opcionalmente, `clientCorrelationId`.
   O backend deduplica por `requestId`, oficializa o fato e então emite o evento com:

   * `eventId` no envelope assíncrono
   * `matchEventId` no payload canônico, se esse campo fizer parte do domínio publicado

2. **`occurredAt` pertence ao payload canônico.**
   Esse é o tempo de domínio.
   O envelope async carrega `recordedAt` como tempo de persistência/publicação do servidor.
   Isso evita confundir “quando aconteceu no jogo” com “quando o sistema registrou”.

3. **Paridade cross-surface não significa igualdade byte a byte.**
   Significa:

   * mesmos campos semânticos do núcleo
   * mesmos tipos canônicos
   * mesmos nomes canônicos
   * envelopes diferentes e obrigatórios por superfície

4. **Visibilidade de campos deve ser filtrada por superfície.**
   O schema canônico pode ter campos com política de exposição.
   O compilador gera projeções seguras por superfície, em vez de despejar o mesmo payload integral em todo consumidor. Isso é coerente com OWASP API3:2023 sobre exposição indevida em nível de propriedade. ([OWASP][2])

Eu endureceria a regra assim:

```yaml
rules:
  - id: "HB-CROSS-SURFACE-001"
    severity: "MUST"
    when:
      all:
        - x-hb-module == "matches"
        - x-hb-canonical-schema == "match_event_v1"
    then:
      enforce:
        canonical_payload_parity_across_surfaces: true
        identical_semantic_types: true
        identical_field_names: true
        surface_specific_envelope_required: true
        sync_request_id_required: true
        async_event_envelope_required: true
        canonical_occurred_at_owned_by_payload: true
        async_recorded_at_owned_by_envelope: true
        field_visibility_filtering_by_surface: true
    strict: true
    override_allowed: false
```

E quebraria em sub-regras para não deixar nada implícito:

```yaml
rules:
  - id: "HB-MATCHES-IDEMPOTENCY-001"
    severity: "MUST"
    when:
      all:
        - x-hb-module == "matches"
        - x-hb-surface == "sync"
        - x-hb-operation-kind == "action"
    then:
      require:
        - requestId
      enforce:
        request_id_format: "uuid"
        request_id_idempotency: true
    strict: true
    override_allowed: false

  - id: "HB-MATCHES-TEMPORAL-001"
    severity: "MUST"
    when:
      all:
        - x-hb-canonical-schema == "match_event_v1"
    then:
      enforce:
        payload_time_field: "occurredAt"
        envelope_server_time_field: "recordedAt"
        forbid_payload_transport_time_aliases: true
    strict: true
    override_allowed: false

  - id: "HB-MATCHES-VISIBILITY-001"
    severity: "MUST"
    when:
      all:
        - x-hb-canonical-schema == "match_event_v1"
    then:
      enforce:
        surface_visibility_filter: true
    strict: true
    override_allowed: false
```

Abaixo estão os 3 JSON Schemas materializados, já endurecidos para:

* idempotência por `requestId`
* separação entre `occurredAt` e `recordedAt`
* paridade de payload canônico
* visibilidade por propriedade via `x-hb-visibility`
* sufixos canônicos (`Id`, `At`, `Count`)

A decisão correta é: **não tornar `matchEventId` autoridade do cliente** e **não exigir `matchEventId` nem `sequenceNumber` no input de submissão**.

Ou seja, não adotar UUID temporário do cliente como identidade canônica do fato. Isso reabre conflito de autoridade. O cliente já tem `requestId` para idempotência e rastreabilidade. O servidor é quem oficializa o fato e emite:

* `matchEventId` canônico
* `sequenceNumber` canônico
* `processingAt`/`recordedAt`

Correção estrutural:
1. `match_event_v1` **MUST** ser o núcleo canônico **persistido/publicado**
2. a submissão sync **MUST NOT** referenciar diretamente** `match_event_v1`
3. a submissão usa um schema derivado de input, por exemplo `match_event_submission_v1`
4. o resultado sync devolve o mapeamento `requestId -> matchEventId`

Em termos de governança, isso significa:
* paridade cross-surface continua existindo no **núcleo semântico**
* mas a superfície de entrada usa uma **projeção de escrita**
* essa projeção remove campos de autoridade do servidor

Essa é a solução determinística. Não precisa de bypass, não reintroduz alucinação e mantém a autoridade onde ela deve estar.

A regra normativa que falta é esta:

```yaml
rules:
  - id: "HB-MATCHES-AUTHORITY-001"
    severity: "MUST"
    when:
      all:
        - x-hb-module == "matches"
        - x-hb-surface == "sync"
        - x-hb-operation-kind == "action"
        - x-hb-canonical-schema == "match_event_v1"
    then:
      enforce:
        input_schema_projection: "match_event_submission_v1"
        forbid_client_authority_fields:
          - matchEventId
          - sequenceNumber
        require_server_authority_fields_in_result:
          - matchEventId
          - sequenceNumber
    strict: true
    override_allowed: false
```

A decisão sobre autoridade de ID fica, portanto, fechada assim:
* `requestId`: autoridade do cliente para idempotência
* `clientCorrelationId`: autoridade opcional do cliente para rastreio
* `matchEventId`: autoridade exclusiva do servidor
* `sequenceNumber`: autoridade exclusiva do servidor

* OBS:`clockSeconds` e `videoTimestampSeconds` para `number`, não `integer`.

O compilador agora precisa reconhecer dois níveis de schema:
* `canonical_schema` para verdade persistida/publicada
* `input_projection_schema` para submissão sync

Sem essa distinção, qualquer módulo event-first com superfície sync voltará a colapsar no mesmo paradoxo.
