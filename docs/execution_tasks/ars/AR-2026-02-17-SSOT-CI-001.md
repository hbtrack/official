# 1) ACTIVE_PLAN (mínimo, determinístico)

objetivo_final:
  Integrar a geração de SSOT (docs/ssot) ao CI, garantindo que todo job de validação rode
  "gerar SSOT → rodar gates", prevenindo falhas futuras ao ativar required:true.

definition_of_done:
  - CI executa: python scripts/ssot/gen_docs_ssot.py --all
  - CI falha se a geração criar diff em docs/ssot (dirty-tree guard)  [somente se docs/ssot for versionado]
  - CI executa gates SSOT (L0/L1 + L2 UDS) e falha com exit code != 0
  - HB_TRACK_PROFILE.yaml lista o comando aprovado do gerador SSOT
  - Nenhuma alteração manual é cometida em docs/ssot/** e docs/_generated/**

invariantes_globais:
  - MUST_NOT alterar flags required (permanece como está; transição em AR separada)
  - MUST_NOT comitar mudanças de conteúdo em docs/ssot/** (SSOT é gerado; alterações só via gerador)
  - MUST_NOT comitar mudanças em docs/_generated/** (DERIVED; outputs de runtime apenas)
  - Gates MUST ler paths dinamicamente do docs/_canon/HB_TRACK_PROFILE.yaml

gates_obrigatorios:
  - pwsh scripts/gates/SSOT/check_hb_profile.ps1
  - python scripts/gates/SSOT/check_uds_consistency.py

evidencias_necessarias_por_gate:
  - check_hb_profile.ps1:
      - logs do job CI (stdout/stderr)
      - (opcional) artifact: docs/_generated/_reports/gates/gate_profile.log
  - check_uds_consistency.py:
      - logs do job CI (stdout/stderr)

status_atual_do_sistema_ou_componente:
  - SSOT migrado para docs/ssot e protegido como DERIVED via docs/_INDEX.yaml
  - Gerador existe: python scripts/ssot/gen_docs_ssot.py --all
  - Gates existem e PASS localmente (L0/L1 e L2)
  - required está em modo gradual (false) e NÃO deve ser alterado nesta AR

dependencias_ou_prerequisitos_conhecidos:
  - Runner CI precisa de Python (versão do projeto) e dependências instaláveis
  - Runner CI precisa de pwsh para executar wrapper PowerShell

stop_conditions:
  - Se o CI não conseguir executar o gerador SSOT, STOP (não "contornar")
  - Se gates retornarem FAIL, STOP
  - Se a mudança exigir alterar required:true, STOP (fazer AR separada)
  - Se precisar comitar outputs de docs/_generated/** ou docs/ssot/**, STOP (corrigir gerador/determinismo)

# 9) OBSERVAÇÕES

## Por que dirty-tree guard é importante?

Se o gerador SSOT criar mudanças não-commitadas em docs/ssot/, significa que:
1. Desenvolvedor esqueceu de rodar o gerador antes de commitar
2. Gerador não é determinístico (outputs variam entre execuções)
3. Configuração do gerador mudou e SSOT precisa ser regenerado

O guard força disciplina: sempre que mudar inputs do SSOT (schema, routes, etc.), 
o desenvolvedor DEVE rodar `python scripts/ssot/gen_docs_ssot.py --all` e commitar os outputs.

## Por que required permanece false?

required:true bloqueia merges quando SSOT está desatualizado. Isso é desejável em produção,
mas requer estabilidade do gerador e CI. Esta AR estabelece a infraestrutura; a transição
para required:true será feita em AR separada após período de monitoramento.

## Integração com Executor Workflow

O executor_workflow.py pode usar este mesmo padrão:
1. Antes de executar plano: rodar `python scripts/ssot/gen_docs_ssot.py --all`
2. Rodar gates obrigatórios
3. Se gates FAIL → status = BLOCKED
4. Se gates PASS → prosseguir com execução
