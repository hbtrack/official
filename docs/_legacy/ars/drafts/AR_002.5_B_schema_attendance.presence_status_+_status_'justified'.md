# AR_002.5_B — Schema: attendance.presence_status + status 'justified'

**Status**: 🔬 EM TESTE
**Versão do Protocolo**: 1.1.0

## Descrição
LACUNA DETECTADA: PRD Seção 7 (US-001) especifica explicitamente 3 estados de presença: 'Present / Absent / Justified' ('Posso marcar Present/Absent/Justified com um clique'). O schema atual restringe presence_status a apenas 2 valores via CHECK constraint: ARRAY['present', 'absent']. O terceiro estado 'justified' é atualmente codificado como: presence_status='absent' + reason_absence IS NOT NULL. Esta ambiguidade causa dois problemas: (A) queries de relatório precisam de lógica condicional para distinguir 'faltou sem justificativa' vs. 'faltou com justificativa'; (B) o frontend precisa renderizar 3 botões mas o schema suporta apenas 2, gerando inconsistência de validação.

AÇÃO: Criar migration Alembic para ampliar o CHECK constraint da coluna presence_status na tabela attendance. Usar Python (.py).

MUDANÇA NO SCHEMA:
```sql
-- Remover constraint existente:
ALTER TABLE attendance DROP CONSTRAINT ck_attendance_status;
-- Adicionar nova com 3 valores:
ALTER TABLE attendance ADD CONSTRAINT ck_attendance_status
    CHECK (presence_status IN ('present', 'absent', 'justified'));
```

MIGRAÇÃO DE DADOS EXISTENTES:
```sql
-- Promover registros ausentes com reason_absence para 'justified':
UPDATE attendance
SET presence_status = 'justified'
WHERE presence_status = 'absent' AND reason_absence IS NOT NULL AND deleted_at IS NULL;
-- Nota: reason_absence permanece preenchido em 'justified' para detalhe.
```

REGRA DE NEGÓCIO RESULTANTE:
- 'present': atleta compareceu
- 'absent': faltou sem justificativa (reason_absence MUST be NULL)
- 'justified': faltou com motivo formal (reason_absence SHOULD be filled)
Adicionar CHECK para reforçar: CHECK ((presence_status != 'absent') OR (reason_absence IS NULL)) se a regra for formalizada.

NOME DA MIGRATION: 0056_attendance_add_justified_status.py
