Jogos

---

Vamos definir agora como deve funcionar a **`ProfessionalSideBar`** (o menu lateral do sistema HB Track) especificamente para a **rota `/games`**, mantendo todos os padrões do design system e garantindo uma **experiência fluida, eficiente e contextualizada para o usuário profissional**.

---

## 🎯 **Objetivo da `ProfessionalSideBar` na rota `/games`**

A `ProfessionalSideBar` deve:

* Servir como ponto de orientação persistente.
* Reforçar o contexto da equipe ativa.
* Permitir navegação rápida entre módulos complementares.
* Adaptar-se visualmente ao conteúdo da rota de jogos.

---

## 🧭 **UX: Passo a passo da experiência ao clicar em "Jogos"**

### 🔹 **1. Usuário entra no sistema**

* A `ProfessionalSideBar` é carregada com a equipe ativa (usando `useActiveTeam()`).
* O botão “Jogos” já aparece na seção `Eventos`, com o ícone `Gamepad`.

---

### 🔹 **2. Usuário clica em “Jogos” na sidebar**

**Comportamento esperado:**

1. O sistema **verifica se há uma equipe ativa**:

   * Se **não houver equipe ativa**:

     * Abre imediatamente o `TeamSelectModal`.
     * Após a escolha, redireciona para `/games?teamId=...`.
2. Se houver equipe ativa:

   * A rota muda para `/games`.
   * A `ProfessionalSideBar` **destaca** o ícone de Jogos com `AppHighlight`.
   * A aba “Jogos” é marcada como **ativa visualmente**.

---

### 🔹 **3. Sidebar expandida com submenu (se necessário)**

A rota `/games` **não precisa de submenu interno fixo**, pois sua navegação é controlada por abas no conteúdo.

Entretanto, a `ProfessionalSideBar` pode:

* Expandir a seção "Eventos" para mostrar:

  * `Calendário`
  * `Competições`
  * `Jogos` (ativo)
* Exibir tooltip ao hover em estados colapsados.

---

### 🎨 **BLOCO 2: Design UI (Tokens e Componentes)**

#### 🔤 Tipografia

* Fonte mínima: 14px (`--app-font-body`)
* Títulos com `--app-font-title-md` ou `lg`, sem decorativo
* Peso: 600 para seções, 700 para títulos

#### 📐 Layout

* Padding padrão: `--app-space-lg` (16px)
* Cards com `AppCard` (uso obrigatório)
* Layout denso, responsivo, com suporte a múltiplas telas

#### 🎨 Cores e Estados

* Status visual por tipo:

  * `success`: Finalizado
  * `warning`: Agendado
  * `danger`: Cancelado
* Borda lateral por tipo de evento (gol, cartão, etc)

#### 🧩 Componentes obrigatórios

* `<AppCard>`, `<AppTable>`, `<AppTabs>`, `<AppEmptyState>`, `<AppSkeleton>`, `<AppToast>`, `<AppTooltip>`, `<AppModal>`, `<AppDrawer>`, `<AppTag>`, `<AppHighlight>`

#### 🚫 Padrões proibidos

* Nenhum `p-*`, `label` manual ou botão sem `AppButton`
* Nada flutuante sem contexto (ex: modal fora de contexto da rota)
* Nenhuma cor fora do sistema de tokens

### 🧬 **BLOCO 7: Modelagem de Dados e Integrações Backend (API)**

Este bloco define **a estrutura dos dados**, **as operações necessárias via API**, e **como cada interação da rota `/games` se comunica com o backend**, em conformidade com os esquemas encontrados no `database.json`.

---

## 🗃️ **Tabelas principais envolvidas**

### 1. `matches` (jogos)

Campos:

```ts
{
  id: string
  team_id: string
  opponent_id: string
  competition_id?: string
  date: string
  location: string
  score_home?: number
  score_away?: number
  status: 'Agendado' | 'Finalizado' | 'Cancelado'
  notes?: string
  created_at: string
  updated_at: string
}
```

---

### 2. `match_participants` (escalação)

```ts
{
  id: string
  match_id: string
  athlete_id: string
  is_starting: boolean
  position?: string
  notes?: string
}
```

---

### 3. `match_events` (eventos do jogo)

```ts
{
  id: string
  match_id: string
  athlete_id: string
  minute: number
  type: 'Gol' | 'Cartão Amarelo' | 'Cartão Vermelho' | 'Substituição' | 'Outro'
  description?: string
}
```

---

### 4. `opponents` e `competitions`

* Usadas em joins para nomes e metadados.

---

## 🔌 **Endpoints REST (ou GraphQL equivalente)**

### 🔹 Criar Jogo

`POST /matches`

* Body: campos principais do jogo
* Retorno: jogo criado (`match.id`)
* Validações: `team_id`, `opponent_id`, `date`, `location` obrigatórios

### 🔹 Buscar Lista de Jogos

`GET /matches?team_id=xyz&status=Finalizado&date_gte=...&date_lte=...`

### 🔹 Buscar Jogo por ID

`GET /matches/:id`

* Retorna dados completos do jogo + joins

### 🔹 Editar Jogo

`PUT /matches/:id`

* Campos: qualquer um editável

### 🔹 Cancelar Jogo

`PATCH /matches/:id`

* Body: `{ status: "Cancelado" }`

---

### 🔹 Escalação

* `GET /match_participants?match_id=xyz`
* `POST /match_participants`: adicionar atleta
* `PUT /match_participants/:id`: editar atleta
* `DELETE /match_participants/:id`: remover atleta

---

### 🔹 Eventos

* `GET /match_events?match_id=xyz`
* `POST /match_events`: criar
* `PUT /match_events/:id`: editar
* `DELETE /match_events/:id`: excluir

---

### 🔹 Relatórios e Exportações

* `POST /matches/:id/report`: gera PDF
* `GET /matches/:id/stats`: retorna estatísticas agregadas
* `GET /matches/:id/export.csv`: gera CSV

---

### 📦 **Validações e Regras de Negócio**

* Um atleta só pode ser escalado 1x por jogo
* Minuto de evento deve ser entre 0 e 130
* Partida finalizada não pode ser editada, apenas visualizada
* Não é permitido excluir jogo já finalizado (apenas cancelar)
* Apenas `staff` com permissão `can_edit_matches` pode editar jogos
