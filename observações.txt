`_reports/` é o root canônico, então a correção determinística precisa “ancorar” o caso de correção em cima do Evidence Pack canônico que já existe na validação. Hoje há um conflito direto: validação fixa `_reports/audit/<RUN_ID>/…` como Evidence Pack (SSOT operacional), enquanto correção fixa `cases/<CORR_ID>/…` como case folder (SSOT de estado).  

Ajuste mínimo (P0) para ficar determinístico com `_reports/` como root canônico:

1. Root canônico único
   1.1 Evidence Pack (execução de gates) continua exatamente como a validação define:

* `_reports/audit/<RUN_ID>/summary.json`
* `_reports/audit/<RUN_ID>/context.json`
* `_reports/audit/<RUN_ID>/checks/<CHECK_ID>/{stdout.log,stderr.log,result.json}`
  Isso é canônico e não muda. 

1.2 O “case folder” da correção deve ir para dentro de `_reports/` (para não existir 2 roots):

* **ANTES:** `cases/<CORR_ID>/…` 
* **DEPOIS (normativo):** `_reports/cases/<CORR_ID>/…`

2. Estado do caso (SSOT fora do chat) precisa apontar para RUN_ID(s)
   O protocolo de correção é explícito que “chat não é estado” e que `state.yaml` é SSOT do caso. 
   Com `_reports/` como root, esse SSOT vira:

* `_reports/cases/<CORR_ID>/state.yaml`

E o `state.yaml` MUST ganhar campos obrigatórios para amarrar a auditoria (RUN_ID):

* `run_ids_repro: [RUN_ID...]` (onde a falha foi reproduzida)
* `run_ids_verify: [RUN_ID...]` (onde passou pós-patch)
* `primary_run_id: RUN_ID` (o “principal” do caso naquele momento)

Sem isso, você volta ao problema original: o executor “escolhe” quais evidências contam.

3. Correção não deve duplicar logs; deve referenciar e hash-ear o Evidence Pack canônico
   Hoje a correção pede `evidence_manifest.json` com sha256, tamanho, etc. 
   Com `_reports/audit/<RUN_ID>/…` canônico, o manifest do caso deve hash-ear (no mínimo):

* `_reports/audit/<RUN_ID>/summary.json`
* `_reports/audit/<RUN_ID>/context.json`
* `_reports/audit/<RUN_ID>/checks/<ID>/result.json`
  (opcionalmente stdout/stderr se não houver restrição de tamanho/LGPD)

Isso resolve “auditabilidade fora do chat” sem duplicar pastas.

4. CHECK_ID vs GATE_ID: você precisa matar o namespace duplo
   Validação usa `CHECK_ID` em `_reports/audit/<RUN_ID>/checks/<CHECK_ID>/…`. 
   Correção define `GATE_ID` como a chave do registry e dos required_gates. 
   Regra determinística recomendada:

* **GATE_ID == CHECK_ID (string idêntica)**
* o registry `docs/_canon/_agent/GATES_REGISTRY.yaml` é SSOT do “id executável” (se você chamar de gate ou check, tanto faz). 

Se você não unificar isso, o sistema continua ambíguo.

5. Exit codes: validação precisa adotar a taxonomia canônica 0/2/3/4 (ou você perde classificação infra vs bug)
   Correção já tem exit codes canônicos 0/2/3/4. 
   Validação ainda define “0 PASS; !=0 FAIL”. 
   Isso é incompatível com o objetivo de determinismo operacional (principalmente distinguir ERROR_INFRA vs FAIL_ACTIONABLE). O ajuste mínimo é: o runner de auditoria deve normalizar qualquer gate legado para 0/2/3/4 e gravar em `result.json`:

* `exit_code` (0/2/3/4)
* `status` (PASS | FAIL_ACTIONABLE | ERROR_INFRA | BLOCKED_INPUT)

6. .gitignore/retention (você não perguntou, mas é obrigatório decidir se quer SSOT “versionado”)
   Se `_reports/` é o root:

* OU você versiona `_reports/cases/**` (state.yaml, patch_plan.yaml, etc.) e ignora `_reports/audit/**` (grande/PII)
* OU você ignora `_reports/` inteiro e então o “estado SSOT fora do chat” deixa de ser versionado (quebra o próprio protocolo).

O protocolo de correção exige estado “em arquivo versionado”. 
Logo, o caminho determinístico é:

* Git TRACK: `_reports/cases/**`
* Git IGNORE: `_reports/audit/**` (exceto se você quiser armazenar audit no repo, o que costuma ser ruim)

7. 
A) Como o RUN_ID é gerado (e como evita colisão / fica rastreável por CORR_ID). A validação só diz `<RUN_ID>`. 
Regra recomendada: para correções, `RUN_ID MUST conter CORR_ID` (ex.: `RUN-CORR-2026-02-18-001-01`), para linkagem determinística.
B) Se `_reports/audit/` será “prova oficial” por quanto tempo (retenção) e como sanitiza/redige (LGPD). A correção já exige redaction/truncation via script Python. 
C) Quem fecha o caso: você já definiu “HUMAN fecha casos”, mas precisa fixar qual evidência de runtime (PROTOCOLO E) é obrigatória para marcar VERIFIED/CLOSED quando o bug é runtime/staging.
